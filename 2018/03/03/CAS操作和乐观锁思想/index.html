CAS操作和乐观锁思想

Posted on 2018-03-03 | Edited on 2018-05-02 | In Java并发编程 | Views:

CAS操作：
1.CAS（Compare and swap）比较和替换是设计并发算法时用到的一种技术。比较和替换是使用一个期望值和一个变量的当前值进行比较，
如果当前变量的值与我们期望的值相等，就使用一个新值替换当前变量的值。
2.显而易见，CAS需要内存位置（V）、预期原值（A）和新值(B)这些参数，并且需要先进行读取操作，然后比较读取值和待写入值，想等则
做一个写入操作。在多线程环境下，如果CAS不能作为一个原子操作则会导致问题。
3.现在CPU内部已经执行原子的CAS操作（JNI unsafe中compareAndSwapInt方法）。JDK1.5后，java.util.concurrent（J.U.C）.atomic包中的一些原子类可以用来使用CPU中的这些功能，
同时弥补volatile关键字不能保证原子性的问题。

例如AtomicInteger, 自增操作(getAndIncrement)就是一个自旋CAS,int变量是volatile的，具有可见性,底层用CAS比较数值，成功则返回当前值，失败则不停自旋重试：
public class AtomicInteger extends Number implements java.io.Serializable {  

    private volatile int value;  

    public final int get() {  
        return value;  
    }  

    public final int getAndIncrement() {  
        for (;;) {  
            int current = get();  
            int next = current + 1;  
            if (compareAndSet(current, next))  
                return current;  
        }  
    }  

    public final boolean compareAndSet(int expect, int update) {  
        return unsafe.compareAndSwapInt(this, valueOffset, expect, update);  
    }  
}

乐观锁思想：
与悲观锁思想相对（总认为资源不安全，先取锁再访问的保守策略思想，例如RDBMS中查询时加S锁或者X锁，是数据处理的最终保证）。
乐观锁假定数据不会有冲突，所以只在最终写操作提交更新时候检查，假设发生了冲突，则返回错误信息，让用户应用程序决定如何处理。

乐观锁=提交时冲突检查+数据更新

根据以上，CAS操作其实可以被认为是乐观锁的一种实现方式，当多个线程尝试对同一个数据进行写操作时，利用CAS对比读取的值和预期值，同一时间只有一个线程可以成功，
失败的线程可以自旋，直到成功为止。CAS的读取比较类似于乐观锁最终写时候的冲突检查，CAS对比成功后的写入对应乐观锁的数据更新。以上JUC中的ATOMIC类就是一种典型
实现。
