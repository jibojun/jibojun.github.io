Java类加载和双亲委托模型

Posted on 2018-03-08 | Edited on 2018-05-09 | In JVM | Views:

对于HotSpot JVM,默认有三种ClassLoader:

1.BootstrtapLoader：
C++实现与JVM一体。加载核心类。例如%JAVA_HOME%/jre/    lib（rt.jar,resources.jar都在这里），    %JAVA_HOME%/    jre/classes。
跟着JVM一起启动初始化。

2.ExtClassLoader
Java实现，加载扩展库。主要在%JAVA_HOME%/jre/lib/ext中
（例如jaccess.jar）。
被Bootstrap加载，并且设置其为父加载器。

3.AppClassLoader
Java实现，加载完Ext完后加载，设其父加载器为Ext。主要加载    classpath环境变量位置里的类，JAR。

这三个默认Loader按照双亲委托模型从上到下形成父子关系，并不继承，而是下层加载器持有上层加载器引用。
双亲委托模型加载机制：
1.从下而上先到第一个Loader,如无自定义，则先到App,检查自身缓 存，如果已经加载则直接返回。
2.如果找不到则不直接加载，而是委托给父Loader,例如App委托给 Ext。一直向上走到最高层：Bootstrap。
3.当所有父加载器都没有从目标路径加载时候，再由当前Loader加载 并且缓存。
可以认为是一个从上到下的加载过程，先信任最上层的，让他先尝试加载，不行则下一层继续。

双亲委托的目的：
namespace，要确定一个类不止由类的全名来决定，也由哪个 class loader加载来决定。同名class不同加载器在JVM内也不是
同一个类。
例如Object（全名java.lang.object），加载时会被委托至 最上层Bootstrap，加载后。
假设用户自己定义了全名相同的object类，这时从自定义加载器一直
向上委托到顶层，会按照路径加载JDK自带的Object类而不是用户的。
这样可以保证类加载的安全性，避免恶意代码。
