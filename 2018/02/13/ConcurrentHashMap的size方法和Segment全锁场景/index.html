ConcurrentHashMap的size方法和Segment全锁场景

Posted on 2018-02-13 | Edited on 2018-05-13 | In Java集合类 | Views:

JDK1.5-1.7中，ConcurrentHashMap主要思想都是分段锁，默认16个分段（segment），既并发度1。写操作put,remove,clear会加锁，但一般情况下只会加在数据所在的segment上，不影响别的。

有一种情况会导致所有segment被锁，segment里有count变量（volatile）存自己分段有多少个元素，简单来说所有分段数组的count字段相加就是整个map的size了，但是这是2个操作，拿count（读取时候如果有别的线程改因为volatile会得知）和相加，读取时没有问题，但做加法时候，如果别的线程往里增删元素，则会不准。

保险的办法就是加锁，size方法执行时候不能做写操作，但这样做法类似HashTable，效率偏低。

ConcurrentHashMap计算size主要逻辑就是：
1.连续两次循环读取每个分段个数和相加操作。
2.Segment里有另一个变量modCount，如果有put,remove,clear 执行，则会加1。
3.通过modCount相加结果比较前后两次计算结果是否相同，不同则认 为有修改发生大小发生了变化，那就加锁住所有segment。最后完成再 一个个解锁。

所以如果size方法上述计算过程中出现了大小变化，在这个场景中，最终ConcurrentHashMap会加锁锁住所有Segment。

try {
    for (;;) {
        if (retries++ == RETRIES_BEFORE_LOCK) {
            for (int j = 0; j < segments.length; ++j)
                ensureSegment(j).lock(); // force creation
        }
        sum = 0L;
        size = 0;
        overflow = false;
        for (int j = 0; j < segments.length; ++j) {
            Segment<K,V> seg = segmentAt(segments, j);
            if (seg != null) {
                sum += seg.modCount;
                int c = seg.count;
                if (c < 0 || (size += c) < 0)
                    overflow = true;
            }
        }
        if (sum == last)
            break;
        last = sum;
    }
} finally {
    if (retries > RETRIES_BEFORE_LOCK) {
        for (int j = 0; j < segments.length; ++j)
            segmentAt(segments, j).unlock();
    }
}
