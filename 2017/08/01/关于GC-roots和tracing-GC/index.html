关于GC roots和tracing GC

Posted on 2017-08-01 | Edited on 2018-08-02 | In JVM | Views:
  1. 传统的引用计数法，维护一个当前对象的引用，计数为0则说明没有引用需要回收。但是有很大弊端，例如引用类型分为强引用，软引用，弱引用，虚引用。且假设两个对象a和b互相持有对方的引用，则每个计数都是1，认为不应该被回收。
  2. JVM使用可达性分析方法，既分析对象是否与GC ROOTS有引用链。
  3. GC ROOTS可以理解为一组必须活跃的引用，对象如果无法通过tracing最后和GC ROOTS相连，或者说从GC ROOTS往下搜索，与对象直接并没有引用链，则可以认为被回收的。
  4. GC ROOTS可能有以下：
    1. JVM中线程栈里的栈帧指向堆里对象的引用，既正在被调用方法中局部变量，参数。
    2. class中static属性成员的引用
    3. 常量，static final
    4. JNI本地方法的引用
  5. tracing root就是从GC ROOTS这个集合出发向下搜索，引用链可达的对象全部认为存活，其余对象则认为无用，可以被回收。
  6. 因为大多垃圾收集器是分代收集，例如ParNew+CMS，一个负责年轻代，一个负责老年代。年轻代MINOR GC时，年轻代是收集部分，老年代是非收集部分，但是可能老年代有指向年轻代对象的引用，也需要被考虑进去。此时，老年代指向年轻代的引用会被考虑成GC ROOTS。
  7. 而老年代触发的FULL GC是针对整个堆，这些则可能不会作为GC ROOTS。所以MINOR GC的GC ROOTS集合则可能多于FULL GC的GC ROOTS。
